2.4. Giải thích cách áp dụng

Command Pattern — RegisterUserCommand

  • Mục đích: Đóng gói toàn bộ thao tác đăng ký (validate → tạo user → gửi OTP) thành một command riêng biệt.
  • Lợi ích: Tách nghiệp vụ đăng ký ra khỏi Controller, dễ test đơn vị, dễ mở rộng (ví dụ: thêm ghi audit log).
// RegisterUserCommand.php
final class RegisterUserCommand
{
    public function execute(RegisterUserDto $dto): void
    {
        $this->validationChain->validate($dto);   // Chain of Responsibility
        DB::transaction(function () use ($dto) {
            $user = $this->userRepository->create([...]);
            $this->otpService->generate($user, OtpContext::REGISTER);  // Adapter
        });
    }
}

Chain of Responsibility — RegistrationValidationChain

  • Mục đích: Kiểm tra tuần tự: email hợp lệ → mật khẩu ≥ 8 ký tự → email chưa tồn tại (Active).
  • Lợi ích: Dễ thêm rule mới mà không sửa code cũ.
// RegistrationValidationChain.php
final class RegistrationValidationChain
{
    public function validate(RegisterUserDto $dto): void
    {
        if (! filter_var($dto->email, FILTER_VALIDATE_EMAIL)) { throw ... }
        if (mb_strlen($dto->password) < 8) { throw ... }
        $existing = $this->userRepository->findByEmail($dto->email);
        if ($existing && $existing->status === UserStatus::Active) { throw ... }
    }
}

3.4. Giải thích cách áp dụng

Strategy Pattern — AuthStrategy

  • Mục đích: Cho phép chuyển đổi thuật toán đăng nhập tùy phương thức (Email/Password hoặc Google OAuth) mà không sửa code chung.
  • Interface: Định nghĩa 2 method method() (loại AuthMethod) và authenticate() (xác thực, trả AuthResultDto).
// AuthStrategy.php
interface AuthStrategy {
    public function method(): AuthMethod;
    public function authenticate(AuthPayload $payload): AuthResultDto;
}

Factory Method — AuthStrategyResolver

  • Mục đích: Tự động chọn Strategy phù hợp dựa trên AuthMethod (enum).
  • Cách hoạt động: Duyệt qua danh sách strategy đã đăng ký, trả về strategy có method() trùng khớp.
// AuthStrategyResolver.php
final class AuthStrategyResolver {
    public function resolve(AuthMethod $method): AuthStrategy {
        foreach ($this->strategies as $strategy) {
            if ($strategy->method() === $method) return $strategy;
        }
        throw new LogicException("No auth strategy for [{$method->value}].");
    }
}

Adapter Pattern — GoogleSocialiteAdapter

  • Mục đích: Bọc dữ liệu từ SDK Socialite (bên thứ 3) thành interface SocialUserAdapter nội bộ.
  • Lợi ích: Khi Socialite thay đổi API → chỉ sửa adapter, không ảnh hưởng business logic.
// GoogleSocialiteAdapter.php
final class GoogleSocialiteAdapter implements SocialUserAdapter {
    public function __construct(private readonly SocialiteUser $socialiteUser) {}
    public function getEmail(): string { return $this->socialiteUser->getEmail(); }
    // ... các phương thức khác
}

4.1. Tổng quan Pattern áp dụng

4.3. Hiện trạng code

Hiện tại, chức năng xem thông tin tài khoản được xử lý trực tiếp trong Controller sử dụng UserService interface và implementation. Logic phân quyền được xử lý qua Laravel Middleware và Policy.

[!TIP] Gợi ý cải tiến: Áp dụng Proxy Pattern tại tầng Service để kiểm tra quyền truy cập (chủ tài khoản hay admin), và Facade Pattern để gom dữ liệu profile, thống kê tin đăng, lịch sử giao dịch vào một điểm gọi duy nhất.

7.4. Giải thích cách áp dụng

Chain of Responsibility — ForgotPasswordChain

  • Mục đích: Xử lý tuần tự các bước quên mật khẩu: Tìm user → Gửi OTP → Ghi log.
  • Cấu trúc: Mỗi handler kế thừa AbstractForgotPasswordHandler, có con trỏ next để chuyển tiếp.
  • Lợi ích: Dễ thêm bước mới (ví dụ: RateLimitHandler, BlockedUserHandler) mà không sửa code cũ.
// AbstractForgotPasswordHandler.php
abstract class AbstractForgotPasswordHandler implements ForgotPasswordHandler {
    private ?ForgotPasswordHandler $next = null;

    public function setNext(ForgotPasswordHandler $handler): ForgotPasswordHandler {
        $this->next = $handler;
        return $handler;
    }

    protected function next(ForgotPasswordContext $context): void {
        $this->next?->handle($context);
    }
}

8.4. Giải thích cách áp dụng

Strategy Pattern — VerificationStrategy

  • Mục đích: Cho phép thay đổi phương thức xác thực (manual, auto, AI) mà không sửa code gọi.
  • Mở rộng: Có thể thêm AutoVerificationStrategy, AIVerificationStrategy trong tương lai.
// VerificationStrategy.php
interface VerificationStrategy {
    public function supports(string $method): bool;
    public function verify(Listing $listing, array $documents): VerificationResult;
}

// ManualVerificationStrategy.php
final class ManualVerificationStrategy implements VerificationStrategy {
    public function supports(string $method): bool { return $method === 'manual'; }
    public function verify(Listing $listing, array $documents): VerificationResult {
        return new VerificationResult(
            isValid: count($documents) > 0,
            reason: count($documents) > 0 ? null : 'No verification documents provided.',
        );
    }
}

9. Chức năng Tin đăng yêu thích

9.1. Tổng quan Pattern áp dụng

Pattern Nhóm Trạng thái Thành phần

10.4. Giải thích cách áp dụng

Command Pattern — UpgradeListingCommand

final class UpgradeListingCommand {
    public function execute(Transaction $transaction, Listing $listing, Package $newPackage, UpgradeContext $context): Listing {
        $this->policy->assertEligible($context);                          // Specification
        $expiresAt = $this->expiryFactory->make($context)->calculate($context);  // Strategy
        $benefitStrategy = $this->benefitFactory->make($newPackage);       // Factory + Strategy
        DB::transaction(function () use (...) {
            $transaction->update(['status' => 'SUCCESS']);
            $listing->update(['package_id' => $newPackage->id, 'package_expires_at' => $expiresAt]);
            $benefitStrategy->apply($context);                             // Strategy
        });
        ListingPackageUpgraded::dispatch(...);                            // Observer
        return $updated;
    }
}

Specification Pattern — CanUpgradeSpecification & CanRenewSpecification

  • Mục đích: Quyết định user có đủ điều kiện nâng cấp hay gia hạn.
  • CanUpgradeSpecification: Kiểm tra gói mới có priority cao hơn gói hiện tại.
  • CanRenewSpecification: Kiểm tra user đang gia hạn gói cùng loại.

Thuật toán Default — Package Score

Chiến lược mặc định tính điểm dựa trên công thức:

final_score = score × multiplier × (1 / (1 + hours/24)) × exp(-decay_rate × hours)
  • score: Điểm cơ bản của tin đăng
  • multiplier: Hệ số nhân từ gói tin (VIP có multiplier cao hơn)
  • decay_rate: Tốc độ giảm điểm theo thời gian
  • Sắp xếp theo pkg_priority DESC trước, rồi final_score DESC

[!IMPORTANT]

Điểm mạnh

  • Strategy Pattern được áp dụng xuất sắc trong 3 module: Auth (đăng nhập), Sorting (sắp xếp), và Verification (xác thực). Mỗi module đều có interface rõ ràng, factory tạo strategy, và nhiều implementation cụ thể.
  • Chain of Responsibility được triển khai bài bản cho Quên mật khẩu với đầy đủ Handler interface → AbstractHandler → Concrete Handlers, đúng chuẩn GoF.
  • Nâng cấp gói tin kết hợp 7 pattern (Command + Strategy×2 + Factory×2 + Specification + Observer), là ví dụ mẫu hoàn hảo cho báo cáo.
  • Kiến trúc Clean Architecture giúp các pattern được đặt đúng tầng.

[!WARNING]

Điểm cần cải tiến

  • Các chức năng Xem/Chỉnh sửa tài khoản và Đổi mật khẩu chưa áp dụng pattern nào, logic nằm trong Service implementation.
  • Bộ lọc tin đăng nên áp dụng Specification + Composite để code query gọn gàng hơn.
  • Chat nên thêm Mediator Pattern để giảm phụ thuộc trực tiếp giữa các participant.
  • Lịch sử giao dịch nên áp dụng State Pattern chính thức (tương tự ListingStatusState) cho Transaction.
